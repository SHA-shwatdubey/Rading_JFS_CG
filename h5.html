🔄 Login Flow — Step by Step (Code Level)

1

User sends: {username: "shashwat", password: "mypass123"}

AuthController.login() call hota hai → AuthService.login() pe jaata hai

2

Database me username search hota hai

repository.findByUsername("shashwat") — agar nahi mila → "Invalid credentials" error

3

Password verify hota hai

Check karta hai: stored password $2a$10$... se start hota hai?
✅ Haan → BCryptPasswordEncoder.matches() se compare
❌ Nahi (purana plain-text) → Direct string compare. Agar match hua → password auto-migrate to BCrypt!

4

JWT Token generate hota hai

jwtService.generateToken(user) → Token me hota hai: {userId, username, role, expiry}

5

Response return hota hai

AuthResponse(token="eyJhbGciOi...", role="USER", message="Login successful")

⚠️ Security Rule: Signup me sirf USER, SUPPORT, MERCHANT role allowed hai. ADMIN role khud nahi le sakte — privilege escalation attack se bachne ke liye!

🔑 JWT Token Structure — Andar Kya Hota Hai?

JWT token ko decode karo (jwt.io pe) → 3 parts milte hain:

🔴 HEADER

{"alg":"HS256","typ":"JWT"}

Algorithm: HMAC SHA-256

🟣 PAYLOAD

{"sub":"shashwat",
 "userId":5,
 "role":"USER",
 "iat":1711720000,
 "exp":1711806400}

24 hour expiry

🟢 SIGNATURE

HMACSHA256(
  base64(header) + "." +
  base64(payload),
  SECRET_KEY
)

Tamper-proof!

📱 OTP System — Kaise Kaam Karta Hai?

Generate OTP:

  • POST /api/auth/otp/generate with {"target":"user@email.com"}
  • 6-digit random OTP generate → DB me save with 5-min expiry
  • Response: {"message":"OTP sent successfully"}

Verify OTP:

  • POST /api/auth/otp/verify with {"target":"user@email.com","otp":"482916"}
  • DB se match + expiry check → valid? → {"verified":true}
  • Expired/wrong? → "Invalid or expired OTP"

🔄 Wallet Top-up Flow — Paisa Kaise Add Hota Hai?

1

Daily Limit Check

Aaj user ne kitna topup kiya? Ledger se calculate: SUM(amount) WHERE type='TOPUP' AND today. Agar ₹50,000 se zyada hoga → "Daily topup limit exceeded"

2

Account Find/Create

findByUserId() — agar user ka wallet nahi hai → naya create with balance ₹0

3

Balance Update

balance = balance + ₹500. @Transactional guarantee — agar kuch fail hua → rollback!

4

Ledger Entry Create

Naya LedgerEntry(type=TOPUP, amount=500, reference=UUID) — ye KABHI delete nahi hota. Permanent record!

5

Events Publish (RabbitMQ)

📤 wallet.events.queue → {userId=5, type=TOPUP, amount=500}
📤 Notification: SMS "Wallet credited ₹500" + Email "Wallet credited successfully"

🔄 P2P Transfer Flow — Ek User Se Doosre User Ko Paisa

1

Validation

Sender ≠ Receiver? ✅ Amount > 0? ✅ Daily transfer limit (₹25,000) exceed nahi? ✅ Daily count (10) exceed nahi? ✅

2

Receiver Verify (Feign → User-KYC)

userClient.getUserById(toUserId) — agar 404 → "Receiver user not found"

3

Balance Check

Sender ke paas ₹500 hai? Agar balance < amount → "Insufficient wallet balance"

4

Money Move!

Sender: balance -= 500 | Receiver: balance += 500
Ledger: TRANSFER_DEBIT (-500) for sender + TRANSFER_CREDIT (+500) for receiver

5

Notifications

📱 SMS to sender: "Transfer successful. Amount sent: ₹500"
📱 SMS to receiver: "Amount received: ₹500"

📖 CQRS Implementation — Wallet me Kaise?

Wallet Service me 2 alag controllers hain — ek likhne ke liye, ek padhne ke liye:

✏️ COMMAND Side (Write)

WalletCommandController → /wallet/update

  • PUT request with WalletUpdateCommand
  • WalletCommandHandler.handle() → DB write
  • Redis cache evict → RabbitMQ event publish
  • Response: 202 Accepted (async!)

📖 QUERY Side (Read)

WalletQueryController → /wallet/query/**

  • GET request for balance/history
  • WalletQueryHandler.handle() → Redis check
  • Cache hit? → instant return (< 1ms!)
  • Cache miss? → DB read → cache store → return
💡 Cache Invalidation: Jab COMMAND side kuch likhta hai → wallet.cqrs.events.queue pe event jaata hai → QUERY side apna Redis cache clear kar deta hai. Next read pe fresh data DB se aayega!

🔑 Idempotency — Duplicate Payment Kaise Rokta Hai?

Socho user ne "Pay" button 2 baar click kar diya. Bina idempotency ke paisa 2 baar kat jaata!

1

Har request me ek unique idempotencyKey hota hai

Frontend ek UUID generate karta hai har payment ke liye — jaise "pay-abc-123"

2

Service pehle check karti hai: "Ye key DB me hai?"

transactionRepository.findByIdempotencyKey("pay-abc-123")

3

Agar mili → Same transaction return karo (no duplicate!)

Pehle check hota hai ki payload same hai → agar same hai → old response return. Agar different hai → IdempotencyConflictException!

🔄 Saga Pattern — Payment Flow (4 Steps + Automatic Rollback)

Saga Pattern ka matlab: "Agar beech me kuch fail ho jaaye, toh jo kiya hai wo undo karo!"

1

Step 1: CreateTransaction

DB me naya Transaction(status=PENDING, type=PAYMENT) save karo

⟲ Agar fail: Transaction status = FAILED mark karo

2

Step 2: ValidateAndReserve

Feign call → Sender valid hai? Receiver valid hai? Sender ke paas balance hai?

⟲ Agar fail: Kuch nahi (read-only step tha)

3

Step 3: AccountingEntries

DEBIT ledger entry sender ke liye + CREDIT ledger entry receiver ke liye. Transaction status = SUCCESS

⟲ Agar fail: Reverse ledger entries banao

4

Step 4: PublishRewardEvent

RabbitMQ pe event → Rewards Service points degi. Transaction history event bhi publish.

⟲ Agar fail: Rewards rollback log karo

❌ Failure Recovery: SagaOrchestrator.compensate() REVERSE order me chalta hai — last completed step se pehle wale tak. Har step ka undo hota hai!

Token Extract

🧩 Design Patterns — Advanced Concepts
CQRS, Saga, Event-Driven, Idempotency — simple language me samjho.

📖 CQRS — Command Query Separation

Simple: "Likhne ka raasta alag, padhne ka raasta alag!"

Write (Command): /wallet/cmd/** → PostgreSQL me write → Redis cache evict → RabbitMQ event publish

Read (Query): /wallet/query/** → Redis se padho (fast!) → cache miss? → DB se padho

Benefit: Reads bahut fast (Redis), writes consistent (PostgreSQL). Cache auto-invalidate via RabbitMQ events.

🔄 Saga Pattern — Distributed Undo

Simple: "Agar beech me kuch fail ho — sab undo karo!"

SagaOrchestrator steps execute karta hai. Har step ka ek compensate() method hai. Failure pe — reverse order me sab compensate!

Use: Transaction Service ka Payment flow. 4 steps + 4 compensations.

📨 Event-Driven — Fire & Forget

Simple: "Kaam ho gaya? Sabko bata do, wait mat karo!"

RabbitMQ queues: 4. Listeners: 6. Wallet topup → event → Rewards points + Transaction log + Email notification. Sab async!

🔑 Idempotency — Duplicate Protection

Simple: "Do baar button dabaya? Ek baar hi charge hoga!"

Har request me unique idempotencyKey. DB me pehle check — agar key exists → old result return. Redis setIfAbsent bhi use hota hai distributed lock ke liye.
